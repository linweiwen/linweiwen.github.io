#一、内存优化

##垃圾回收机制

###什么是垃圾回收

  • 麦卡锡John McCarthy 1959年提出
  • 首先在Lisp语言实现垃圾回收机制
  • 对比C/C++这种需要自己管理内存的语言(我们需要自己为对象申请内存空间并归还，稍有不慎就会造成内存泄漏。)
  • 由此，就有了自动内存管理和回收机制
  • 垃圾回收器负责回收程序中已经不使用，但仍然被各种对象占用的内存
  • 将程序员从繁重，危险的内存管理工作中解放出来

缺点：可能会占用大量资源

###关于垃圾回收
（Android本身使用Java语言，具有垃圾回收机制。）
PPT内容 - Android平台最吸引开发者的一个特性：
有垃圾回收机制，无需手动管理内存，
Android系统会自动跟踪所有的对象，并释放那些不再被使用的对象。

###Android中的垃圾回收机制

虚拟机会把系统的区域，划分为三个区域：

  1. Young Generation （新生代）
  2. Old Generation （老生代）
  3. Permanent Generation (永久代)

Young放生命周期比较短的，Old放生命周期比较长的，
永久代是就放一些静态信息。

Young Generation：
又被分为三个区域，Eden/S0/S1 。

  • 大多数新建的对象都位于Eden区
  • 当Eden区被对象填满时，就会执行Minor GC。并把所有存活下来的对象转移到其中一个survivor区。
  • Survivor Space:S0/S1 有两个，存放每次垃圾回收后存活的对象。
  • Minor GC同样会检查survivor区中存活下来的对象，并把它们转移到另一个survivor区。这样在一段时间内，总会有一个空的survivor区。

Old Generation：

  • 存放长期存活的对象和经过多次Minor GC后依然存活下来的对象。
  • 满了进行Major GC。

Permanent Generation：
存放方法区，方法区中有 要加载的类信息/静态变量/final类型的变量/属性和方法信息。

备注：“Minor GC”是比较轻量的GC，执行时间较短;Major GC是比较重的GC，执行时间较长。

###垃圾回收机制&FPS

  • Android系统每隔16ms发出VSYNC信息，触发对UI进行渲染，那么整个过程如果保证在16ms以内，就能达到一个流畅的画面。60FPS。
  • 如果某一帧操作超过了16ms，就会达不到60FPS，就会让用户感到卡顿。
  • (上面提到发生GC时，可能会暂停当前程序执行。)那么，如果UI渲染过程发生GC，导致某一帧绘制时间超过16ms，就达不到60FPS,用户就会觉得卡顿。

###典型场景：内存泄漏

  • 应用程序分配了大量不能被回收的对象
  • 如果分配了大量不能回收的对象，系统可分配内存越来越少了
  • 就会导致新对象的创建需要的内存不够，就会调用GC
  • GC之后再分配，
  • 这样就有可能导致上面FPS达不到60FPS而造成卡顿

##内存检测工具

###三个工具
Android Studio提供了下列工具帮助我们分析应用的内存使用情况：

  • Memory Monitor
  • Allocation Tracker
  • Heap Viewer

###Memory Monitor
可选定要监控的“设备”以及“应用”，
图中蓝色就是应用所用内存，灰色是空闲内存，峰值就是发生了垃圾回收。

###Allocation Tracker
用于跟踪对象内存分配的工具，追踪已分配的内存、创建的对象、对象的数量和它们所占的内存大小，以及这些对象是在哪被创建出来的等等信息。

圆环由里到外，是不同包名所占用内存的大小。

###由Allocation Tracker引入内存抖动的概念

  1. 内存抖动是因为在短时间内大量的对象被创建又马上被释放。瞬间产生大量的对象会严重占用Young Generation的内存区域，当达到阀值，剩余空间不够的时候，会触发GC从而导致刚产生的对象又很快被回收。即使每次分配的对象占用了很少的内存，但他们叠加在一起会增加Heap的压力，从而触发更多其他类型的GC。这个操作有可能会影响到帧率（达不到60FPS），并使用户感知到性能问题。

  2. Allocation Tracker可以很方便地帮我们发现 内存抖动 的现象。

  3. 上机：创建一个线程，线程里执行死循环，循环里不停地创建对象。再放两个按钮，一个启动线程，一个停止线程。

###Heap Viewer

Heap Viewer可以帮我们实时展示应用运行时所有已分配的对象大小、类型信息，
它可以帮我们检测内存泄漏的问题。

上机：正常情况下，旋转屏幕后的Activity能被GC回收掉。但如果有静态变量，就不行了，通过Heap Viewer可以发现这个问题。

###简单总结这三个工具

1.Memory Monitor

  • 方便显示内存使用和GC情况
  • 快速定位卡顿是否和GC有关
  • 快速定位Crash是否和内存占用过高有关
  • 快速定位潜在的内存泄漏问题（那条线会一直增长）
  • 简单易用
  • 不能准确定位问题

2.Allocation Tracker

  • 定位代码中分配对象的类型、大小、时间、线程、堆栈等信息
  • 定位内存抖动问题
  • 配合Heap Viewer一起定位内存泄漏问题
  • 使用复杂

3.Heap Viewer

  • 内存快照信息（实效性好）
  • 每次GC之后收集一次信息
  • 查找内存泄漏利器
  • 使用复杂

###LeakCanary

  • A memory leak detection library for Android and Java
  • https://github.com/square/leakcanary
  • 查找内存泄漏的神器

由square开发的检测内存泄漏的库。
如果发生内存泄漏，会在通知栏发出通知。
把LeakCanary添加到我们的项目，只要在Gradle加两句，
并在Application类的onCreate方法中install。

#二、视图优化

###Why 60FPS
60FPS或16ms每帧。
60FPS ： 1秒显示60帧画面。

为什么是60FPS？
要从大脑说起，它可以将连续的静态画面产生动态的效果，
要让大脑能感受到动画的效果，至少要达到10～12帧每秒，
如果要让大脑感受频繁动画的效果，至少要24帧每秒，这对大脑来说已经非常连续，
当帧率达到60帧每秒，已经是基本上最流畅的了。

###VSYNC
刷新率：硬件刷新频率
Android中大部分情况就是屏幕的刷新频率。

帧率：GPU一秒内可以绘制的帧数。

帧率和刷新率一起配合，才能保证我们应用完美地渲染出来。

GPU会源源不断地朝缓冲区中存入图像数据，屏幕也会定期、持续从缓存区中读取图像。

GPU产生图像的速率是大于屏幕产生图像的速率的。

###真实例子
Google办公室一角的照片。
撕裂现象，整张图片有一个割裂的效果。
GPU在图例图像数据时，会将数据结果按一行一行将图像数据从上到下写入图像，
而屏幕读取时候时，是不知道图像是否已经写入完了，它是整张的区域去读取的，

###双缓冲机制

GPU把缓冲数据写入到Back Buffer区域中，
而屏幕读取数据则是从Frame Buffer区域读取，
Back Buffer会定期地将数据同步到Frame Buffer中，
VSYNC可以保证屏幕在读取Frame Buffer时，不会将Back Buffer的数据写入到Frame Buffer，
这样就可以解决上面的撕裂现象。

###帧率<刷新率
如果屏幕的刷新频率大于帧率，
这样就会出现屏幕两次刷新中，只渲染了一帧的图像，
因此我们要保证GPU渲染足够快。

#四、Bitmap优化

##复用Bitmap

###Bitmap decode

  • 常见的jpg、png、webp是图像的存储格式
  • Android中要显示图片必须先经过解码（decode）读取图像的数据到内存中
  • BitmapFactory提供了常用的一些decode方法
  • 图片真正占用的内存大小要看decode之后的数据大小

1.图像存储格式
jpg是有损压缩格式，当图片色彩比较丰富时，压缩效果就比较明显。
png是为了取代gif，采用无损压缩格式。
webp是Google近几年推出的格式，可以在采用无损压缩的情况下，极大地压缩体积。

2.解码
解码就是将图像数据读取到内存，然后转换成GPU能读取的格式。

3.Android使用BitmapFactory进行解码

###Bitmap memory
位图解码后的大小，在Android不同版本，其实是有所不同的。

<=2.3.3(api 10)

  • Bitmap解码之后的数据存储在Native Memory中
  • 手动调用Recycle回收

也就是我们要手动调用Recycle方法去回收这块内存，
因为这块内存是在C++那一层的，没有Java的垃圾回收机制。

>=3.0(api 11)

  • Bitmap解码之后的数据存储在Dalvik heap中
  • Dalvik自动回收

这里就有了垃圾回收机制，自动帮我们判断Bitmap内存的情况。

###Re-using Bitmaps
Bitmap成为可能导致内存抖动的元凶。

从上图可以，每当要一个Bitmap对象要创建，
因为它需要一块连续的内存空间才能显示，
如果当时不够内存，系统就进行垃圾回收腾出空间，
一旦频繁产生Bitmap，就有可能产生内存抖动，造成卡顿。
那么，如何避免这种现象呢？

使用对象池去复用对象，这样就不用系统每次分配一块新的内存。

所谓对象池（Object pool），
其实就是一块内存缓冲区间，程序启动时预先向Java虚拟机申请一块区域，
用这块区域当缓冲区间来当对象池使用。

接下来所有某一类型对象的创建，都是放到这块内存中，
一旦这块内存被占满，新来的会按照算法淘汰掉老的、不用的旧对象，
这样就避免了在创建信的对象区域。

好处：避免了JVM再进行内存分配及可能产生的垃圾回收。

包装类型比基本类型所占空间要大，不注意的话，

###NoBoxing containers：

容器 键值对类型
HashMap <obj,obj>
SparseBoolMap <bool,obj>
SparseIntMap <int,obj>
SparseLongMap <long,obj>
LongSparseMap <long,obj>

上面那些没有Autoboxing的容器，其实是基于ArrayMaps基础上实现的。
ArrayMaps它的key所占的空间比HashMap的key空间小很多。

###SparseArray

使用SparseArray最大的好处，就是我们可以用索引来遍历，
而不需要像HashMap那样用iterator来遍历。

Android提供了一系列SparseArray

  • android.util.SparseArray
  • android.util.SparseBooleanArray
  • android.util.SparseIntArray
  • android.util.SparseLongArray

但并不是什么时候都适合用SparseArray的，
使用SparseArray最主要适合下面两个场景：

  1. 对象比较少，一般少于1000个（会频繁对容器内元素进行访问，越频繁越显得高效）
  2. 容器中嵌套容器（例如HashMap中又需要有HashMap的情况）

###HashMap memory

HashMap的内存分配情况：
HashMap在创建时首先会分配比较大的数组空间，
用来存放key的hash值。即使它没有存放任何元素，
这块区域其实已经浪费了空间了。

而ArrayMap则没有这样的情况。

总之，容器方面就是使用ArrayMap和SparseMap。

##枚举的使用

我们要注意枚举的使用，它对我们应用的影响主要有两个方面：

  1. 体积的影响
  2. 运行时对性能的影响

例如如果三个变量，用int类型增加124字节，而用枚举则是用int类型的13倍。
Android启动时是很关注dex的大小的，dex体积越大，所占内存更高。

另外，使用枚举，运行时所需内存也会大大增多。

枚举的好处，就是可以减少类型错误，不需要我们额外进行检查，
可以让编译器帮我们去检查。
